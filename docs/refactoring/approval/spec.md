# 전자결재(Approval) 도메인 리팩토링 - Spec

## 배경
LOG-IN의 전자결재 기능은 초기 구현 이후 신규 요구사항이 단일 서비스 클래스에 누적되며,
유지보수와 확장이 어려운 상태에 도달했다. 이미 리팩토링이 완료된 휴가(Leave) 도메인이
명확한 책임 분리와 도메인 모델 중심 설계를 따르고 있는 반면, 결재 도메인은 그 이전의
구조를 그대로 유지하고 있어 일관성이 떨어진다.

본 리팩토링은 결재 도메인을 휴가 도메인과 동일한 수준의 구조적 일관성과 도메인 모델
표현력을 갖추도록 정리하는 것을 목표로 한다.

## 해결하려는 문제

### 1. God Class 문제
`ApprovalService`가 1,066줄에 달하며 결재 CRUD, 파일 처리, 부서/멤버 조회 등
이질적인 책임이 한 클래스에 혼재한다. 단위 테스트 작성이 어렵고, 한 기능의 변경이
다른 기능에 영향을 미칠 위험이 높다.

### 2. 컨트롤러 비대화
`ApprovalController`가 470줄이며, 결재번호 생성·번호 조립과 같은 비즈니스 로직이
컨트롤러에 포함되어 있다. 컨트롤러 본연의 역할(요청 위임, 응답 구성)을 넘어선다.

### 3. 상태 관리 불명확
결재 상태가 한글 문자열("처리 중", "승인", "반려", "회수", "임시저장")로 관리되어
오타 위험이 상존하고, 상태 전이 규칙이 코드 어디에도 명시되어 있지 않다. 잘못된
전이(예: 이미 승인된 결재를 다시 반려)를 막을 장치가 없다.

### 4. 공통 패턴 미준수
프로젝트 전체가 `ResponseMessage<T>` / `ErrorResponse` 기반의 공통 응답 체계를
따르도록 정비되었으나, 결재 도메인만 자체 `ResponseDTO`를 사용한다. 또한 에러 처리도
`try-catch` 후 `null` 반환 방식으로, `BusinessException` 기반 처리와 불일치한다.

## 목표

### 도메인 모델 측면
- 결재 상태와 결재자 상태를 Enum으로 명시하고, 상태 전이 규칙을 Entity 내부에
  메서드로 표현한다.
- 잘못된 상태 전이 시도는 도메인 계층에서 차단되어야 한다.

### 구조 측면
- 결재 서비스의 책임을 명령(Command)과 조회(Query)로 분리한다.
- 파일 처리와 번호 채번 같은 부수적 책임을 전담 컴포넌트로 분리한다.
- 컨트롤러는 요청 위임 역할만 수행하도록 슬림화한다.

### 일관성 측면
- 휴가 도메인과 동일한 패턴을 적용한다.
- 공통 응답·에러 체계로 통합한다.

## 성공 기준

| 지표 | Before | After (목표) |
|---|---|---|
| ApprovalService 라인 수 | 1,066줄 | Command/Query 분리, 각 200~400줄 |
| ApprovalController 라인 수 | 470줄 | 100~120줄 |
| 상태값 표현 | 한글 문자열 | Enum 100% |
| 응답 체계 | 자체 ResponseDTO | ResponseMessage<T> / ErrorResponse |
| 상태 전이 검증 | 없음 | Entity 내부 메서드 (canTransitionTo) |
| 에러 처리 | try-catch + null 반환 | BusinessException 위임 |

추가로:
- 모든 7단계가 각 단계 완료 시점에 **독립적으로 빌드 가능**한 상태를 유지한다.
- 기존 API 명세(요청/응답 JSON 구조)는 변경되지 않는다.

## 범위 외 (Out of Scope)
- 결재 비즈니스 정책의 변경 (결재선 규칙, 회수 가능 조건 등)
- 프론트엔드 변경
- DB 스키마 변경 (컬럼 타입은 VARCHAR 유지, 휴가 도메인과 동일하게
  `@Enumerated(EnumType.STRING)` 방식으로 영어 코드 저장)
- 단, 기존 DB에 한글 값으로 저장된 데이터는 영어 코드로 마이그레이션
  필요 (단계 1에서 마이그레이션 스크립트 함께 작성)
- 새로운 기능 추가
- 자동화된 테스트 작성

## 비기능 요구사항
- 결재 도메인은 자동화된 테스트가 없으므로, 각 단계 완료 시점에 핵심 API
  (기안 작성, 결재 처리, 회수, 조회) 동작을 Postman 등으로 수동 검증한다.
- API 응답 형식이 외부적으로 동일해야 한다 (내부 구현만 변경).
- 향후 테스트 코드 추가는 별도 작업으로 진행한다 (본 리팩토링 범위 외).

## 코드 분석으로 발견된 추가 문제

기존 `ApprovalService`와 `ApprovalController` 분석 과정에서 단순 구조 문제 외에
비즈니스 룰 결함과 보안 결함을 발견했다. 본 리팩토링은 이 결함들도 함께 해결한다.

### 🚨 보안 결함 (단계 1·7에서 우선 해결)

**[I] memberId 헤더 백도어**: 기안 API가 `@RequestHeader("memberId")`로 사용자 ID를
받고, 헤더 값이 있으면 SecurityContext를 우회한다. 인증 시스템을 무력화하는 백도어.
→ 헤더 파라미터 제거. SecurityContext만 신뢰. (단계 7에서 컨트롤러 정리 시 처리)

**[A-확장] 회수 API 인증 정보 부재**: `PUT /approvals/{approvalNo}/status`가 결재번호만
받고 인증 정보를 받지 않는다. 결재번호만 알면 누구나 어떤 결재든 회수 가능.
→ 단계 1에서 `Approval.withdraw(memberId)` 메서드 추가 + 단계 7에서 Controller가
   SecurityContext에서 memberId 추출 후 전달

### 단계 1에서 해결
**[A] 회수 검증 누락**: 주석에는 "다른 사람이 처리했으면 회수 불가"라고 명시되어
있으나 검증 코드가 없다. 본인 확인도 없어 누구나 아무 결재나 회수 가능한 상태.
→ `Approval.withdraw(memberId)` 메서드에서 본인 확인 + 상태 전이 검증 수행
→ "결재자 처리 여부" 검증은 단계 6의 CommandService에서 (Entity가 결재자 컬렉션을
   알 필요 없도록 책임 분리)

### 단계 6에서 해결
**[B] 결재 순서 무시**: 결재 완료 판단이 `i == size - 1` (인덱스 기반) 로만 되어
있어, 앞 결재자들이 처리 안 한 상태에서 마지막 결재자가 먼저 승인하면 전체 승인 처리됨.
→ "모든 결재자가 APPROVED 상태일 때 전체 APPROVED" 로직으로 변경

**[C] 수정이 삭제 후 재생성**: `updateApproval`이 기존 결재를 delete 후 새로 insert.
ID 자체가 바뀌어 이력 추적·외부 참조 불가능. dirty checking 미사용.
→ JPA dirty checking 기반의 정상 수정으로 전환

**[D] 임시저장 status 클라이언트 신뢰**: 기안 시 `approvalDTO.getApprovalStatus()`를
그대로 저장. 클라이언트가 `APPROVED`를 보내면 결재 없이 승인 상태로 시작 가능.
→ 서버에서 신규 결재 status를 강제 결정 (TEMP_SAVED 또는 PROCESSING만 허용)

**[F] save() 호출로 dirty checking 미활용**: Builder로 새 객체 만들어 save 호출.
영속성 컨텍스트의 기존 객체와 충돌 위험.
→ Entity 메서드 호출 + @Transactional dirty checking으로 전환

**[G] 예외 삼키기**: `catch (Exception e) { log.info("오류"); }` 패턴 다수.
트랜잭션 안에서 예외를 삼키면 롤백되지 않아 데이터 정합성 깨짐.
→ BusinessException 위임으로 통일

**[L] 임시저장→기안 전환이 "삭제 후 재생성"**: `POST /approvals` 하나가 신규 기안과
임시저장→기안 전환을 모두 처리. 전환 시 기존 결재를 삭제하고 새 번호로 다시 생성.
결재번호가 바뀌어 이력 추적 불가.
→ 임시저장→기안 전환은 같은 결재번호 유지하며 상태만 변경 (`Approval.submitFromTempSaved()`)
→ Controller에서 임시저장 분기 로직 제거, Service가 책임짐

### 단계 3에서 부분 해결
**[G-부분]** 컨트롤러의 try-catch + null/ResponseDTO 반환 패턴
→ BusinessException 위임 + GlobalExceptionHandler

### 단계 5에서 해결
**[K] 결재번호 생성의 동시성 문제**: 컨트롤러에서 `selectApprovalNo`로 마지막 번호
조회 → 파싱 → +1 → 포맷. 두 사용자가 동시에 기안하면 같은 번호 발급 가능.
→ `ApprovalNoGenerator` 컴포넌트에서 단일 진입점으로 채번 (필요 시 DB 시퀀스 또는
   @Lock 적용 검토)

### 단계 7에서 해결
**[H] 인증 정보 추출 중복**: `SecurityContextHolder.getContext().getAuthentication()
.getName()` 패턴이 컨트롤러 메서드마다 반복.
→ 헬퍼 메서드(`getCurrentMemberId()`)로 추출

**[J] 이모지 로그 폭격**: `🎉🎉🎉🎉🎉` 로그가 운영 환경 가독성 저해.
→ 디버그 로그 정리, 필요한 경우 적절한 로그 레벨로

## 향후 개선 (본 리팩토링 범위 외)
다음 항목은 인지하고 있으나, 본 리팩토링의 범위 관리를 위해 의도적으로 미룬다.

- **[E] 기안자가 Approver 테이블에 `_apr000`으로 자동 등록되는 관례**: 기안자는
  결재 요청자이지 결재자가 아니므로 도메인 모순. 별도 필드 또는 관계로 분리 필요.
  단, DB 스키마 변경을 수반하므로 별도 작업으로 진행.

- **자동화된 테스트 코드 부재**: 단위·통합 테스트 작성은 별도 작업으로 진행.

- **DB 스키마 정규화**: 상태 컬럼이 VARCHAR로 한글/영문이 섞일 수 있는 구조.
  향후 ENUM 타입 또는 코드 테이블로 분리 검토.

- **REST API 명세 정비**: 일부 API가 RESTful 원칙에 맞지 않음 (POST 하나가
  신규 생성과 수정을 모두 처리하는 등). 클라이언트 영향이 크므로 별도 협의 후 진행.
